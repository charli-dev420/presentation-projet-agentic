# Buyer Evaluation / Evaluation acheteur

[FR](#francais) | [EN](#english)

## Francais

### Ce qu'il faut regarder

| Critere | Question |
| --- | --- |
| Maturite | Qu'est-ce qui est demo, prototype, beta ou pret a integrer? |
| Preuves | Quel niveau de claim est atteint et avec quels artefacts rediges? |
| Securite | Quelles donnees restent locales, quelles actions sont explicites? |
| UX | Le resultat est-il lisible par un utilisateur non developpeur? |
| Integration | Quels outils doivent etre connectes et sous quel cadre? |
| Cout | Les mesures sont-elles instrumentees ou seulement estimees? |

### Signaux positifs

- Les claims sont limites par niveau de preuve.
- Les sources privees sont resumees sans fuite.
- Les flux read-only et mutating sont separes.
- Les decisions utilisateur restent explicites.
- Les limites sont documentees avant la demo.

### Questions a poser

1. Quel scenario doit etre prouve en premier?
2. Quelles donnees peuvent etre utilisees?
3. Quel niveau de redaction est requis?
4. Quelles actions doivent demander confirmation?
5. Quel livrable public peut etre publie apres evaluation?

## English

Buyers should evaluate maturity, claim level, security boundaries, UX readability, integration requirements, and cost instrumentation. Strong signals include explicit proof levels, redacted private sources, read/write separation, human decisions, and documented limits.
